# Pinger Relay

## Зачем нужен relay
Pinger связывает телефон и компьютер по трём каналам в порядке приоритета: **Wi-Fi → Bluetooth → relay**. Первые два работают, только когда устройства **рядом** (одна Wi-Fi-сеть или в радиусе Bluetooth).

**Relay — это облачный канал на случай, когда общей сети и Bluetooth нет:** телефон на мобильном интернете, ты ушёл из дома, телефон и ПК в разных сетях. Тогда оба устройства встречаются через relay-сервер в интернете, и связь продолжает работать.

Без настроенного relay Pinger остаётся только локальным (дома/рядом). Свой relay даёт облачную связь **под твоим контролем**, без зависимости от чужого сервера.

> **Это нужно, только если хочешь облачный доступ.** Для связи в пределах дома/одной сети relay не требуется.

## Как это устроено (и почему безопасно)
Relay — «глупая» труба: он сводит две стороны по непрозрачному `roomId` и пересылает между ними **уже зашифрованные** кадры. **Он никогда не видит открытый текст** — сопряжение и сквозное шифрование происходят только между телефоном и компьютером, по секрету, который relay не знает. Поэтому свой relay можно безопасно поднять на любой VPS.

## Что нужно
- VPS с публичным IP.
- Домен (или поддомен) с **DNS A/AAAA-записью на IP сервера**.
- Открытый **TCP-порт 443**.

TLS-сертификат Let's Encrypt берётся **автоматически** (TLS-ALPN-01, порт 80 не нужен).

## Установка — три варианта

### A) Docker на чистой VPS (рекомендуется)
Образ публичный, собирать ничего не надо:
```bash
docker run -d --name pinger-relay --restart unless-stopped \
  -p 443:8443 \
  -e LISTEN_ADDR=:8443 \
  -e TLS_DOMAIN=relay.example.com \
  -v pinger-relay-acme:/var/lib/pinger-relay \
  ghcr.io/wigleg90/pinger-relay:latest
```

### B) Нативно, без Docker (.deb или скрипт)
Скачай из **[Releases](../../../releases)** (тег `relay-vX.Y.Z`) и:
```bash
sudo dpkg -i pinger-relay_*_amd64.deb
sudo sed -i 's/^TLS_DOMAIN=.*/TLS_DOMAIN=relay.example.com/' /etc/pinger-relay/pinger-relay.conf
sudo systemctl enable --now pinger-relay
```
Либо одной строкой (скачивает бинарь + ставит systemd-сервис):
```bash
sudo sh install.sh relay.example.com
```

### C) За Traefik
Если на сервере уже есть Traefik — он терминирует TLS, relay внутри отдаёт plain `ws`. См. `docker-compose.yml` в репозитории сервера.

## Подключение клиентов
В каждом клиенте Pinger (Android / macOS / Windows) укажи адрес своего relay (например `relay.example.com`) в настройках. **Телефон и компьютер должны указывать на один и тот же relay.** Сопряжение — по QR, как обычно.

Проверка сервера: `curl https://relay.example.com/healthz` → `ok`.

## Траблшутинг
- **Сертификат не выдаётся / TLS не поднимается** — DNS ещё не указывает на сервер (подожди распространения) или закрыт порт 443. `TLS_DOMAIN` должен **точно** совпадать с доменом из DNS.
- **`docker pull` не работает** — образ должен быть публичным (по умолчанию так и есть).
- **Не биндится :443 (Docker)** — используй маппинг `-p 443:8443` с `LISTEN_ADDR=:8443` (как в примере выше): контейнер работает непривилегированно и не биндит 443 напрямую.
- **Сервис не стартует (нативно)** — `journalctl -u pinger-relay`; проверь, что задан `TLS_DOMAIN` в `/etc/pinger-relay/pinger-relay.conf` и порт 443 свободен.
- **Клиент не использует relay** — relay включается как fallback, когда нет Wi-Fi/Bluetooth. Убедись, что адрес relay одинаково задан и на телефоне, и на компьютере.
- **Проверка живости** — `curl https://<домен>/healthz` должен вернуть `ok`.
